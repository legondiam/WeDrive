# 重要问答

## 为什么这些业务需要上锁

### 1. 分块合并完成

分块合并会把一次上传会话最终落成用户文件记录，并更新用户空间。

这里需要防止同一个 `upload_session` 被并发合并。如果两个请求同时完成同一个上传会话，可能出现重复创建用户文件、重复更新用户空间、重复修改上传会话状态的问题。

所以这里用了两层保护：

- Redis 分布式锁：先拦住同一个 `sessionID` 的并发合并请求。
- 数据库行锁：事务内锁住 `upload_sessions` 当前行，兜底保证同一条上传会话只能被一个事务修改。

### 2. 秒传

秒传依赖已经存在的 `file_stores` 文件池记录，然后创建一条新的 `user_files` 引用。

这里加锁的重点不是防止用户重复秒传，而是防止秒传新增引用时，另一个事务正在永久删除同一个文件池记录。

如果不锁，可能出现：

1. 秒传准备引用 `file_stores.id = 10`。
2. 另一个请求判断这个文件池没有引用了。
3. 另一个请求删除了 `file_stores.id = 10`。
4. 秒传继续创建 `user_files.file_store_id = 10`。

这样用户文件就会引用到已经被删除的文件池记录。

所以秒传事务里会锁住目标 `file_stores` 行，确认文件池记录稳定后，再创建用户文件引用。

### 3. 永久删除文件

永久删除普通文件时，需要删除用户文件记录、扣减用户空间，并判断文件池是否还有其他引用。

如果没有锁，可能出现：

1. 永久删除事务统计 `fileStoreID` 没有其他引用。
2. 秒传事务同时准备给这个 `fileStoreID` 创建新引用。
3. 永久删除事务删除文件池记录。
4. 秒传事务创建引用，指向已删除的文件池。

所以永久删除时会锁住对应的 `file_stores` 行，再统计引用数量并决定是否删除文件池记录。

总结：这些锁主要保护的是“文件池记录”和“用户文件引用”的一致性，避免并发下出现重复写入、重复扣加空间、引用已删除文件池等问题。

## 为什么秒传准备态要原子读取删除？

秒传的 `prepare_id` 本质上是一次性挑战凭证。`PrepareInstantUpload` 会生成随机片段挑战，并把挑战状态保存到 Redis；`InstantUpload` 再带着 `prepare_id` 和证明回来完成秒传。

如果这里使用普通的两步操作：

```text
GET instant_prepare:{prepareID}
DEL instant_prepare:{prepareID}
```

在高并发下会有竞态窗口。两个请求可能同时 `GET` 到同一个准备态，然后都完成证明校验，最后都进入落库流程。这样 `prepare_id` 就不是严格的一次性凭证。

所以这里使用 Lua 把读取和删除合成 Redis 内部的一次原子操作：

```lua
local value = redis.call("GET", KEYS[1])
if value then
    redis.call("DEL", KEYS[1])
end
return value
```

这样第一个请求会拿到准备态并立即删除 key，第二个并发请求再来时就拿不到准备态，只能返回 `ErrInstantPrepareInvalid`。

这里不使用分布式锁，是因为要解决的问题不是“同一时间只能一个请求执行”，而是“这个 `prepare_id` 只能被消费一次”。原子消费比加锁更贴合语义，也避免了锁 TTL、续期和释放锁带来的额外复杂度。
